## Bearer-token auth

By default, every request must carry an `Authorization: Bearer <token>` header. The token is a 22-character base62 string (~131 bits of entropy) generated by the iOS app on first server enable and stored in `@AppStorage`.

```bash
curl -H "Authorization: Bearer oRjUmrdnbrK8reAtGpDIvq" http://192.168.1.244:5606/workouts
```

- Tap the token in Settings → it's copied to the clipboard.
- Tap **Regenerate Token** → a confirmation alert, then a fresh token replaces it atomically — the LAN server hot-swaps the auth without bouncing the listener, so consumers using the old token start getting `401`s immediately and the URL row never disappears.
- Toggling **Require auth** off in Settings makes the server accept unauthenticated requests. Useful for `curl` exploration on a trusted home network; not recommended otherwise.

When auth is required but the request fails it:

```http
HTTP/1.1 401 Unauthorized
Content-Type: application/json
WWW-Authenticate: Bearer realm="Wirebody"

{"error":"unauthorized"}
```

## Foreground-only execution

The LAN server is built on Apple's `Network` framework (`NWListener` over TCP). It **only runs while the Wirebody app is in the foreground.**

This is an iOS platform limitation, not a Wirebody design choice. iOS suspends third-party apps within ~30 seconds of being backgrounded; arbitrary listening sockets are not preserved. Apple offers narrow background modes (VoIP, audio, location) for specific app categories — none of which legitimately apply to a generic LAN server, and abusing them is an App Store rejection.

In practice:

- Open Wirebody, leave it on screen → the server is up.
- Switch apps, lock the phone, or use FaceTime → iOS suspends Wirebody within ~30 seconds → the listener stops accepting new connections.
- Open Wirebody again → the server comes back on the same port within 1-2 seconds.

The Settings screen says **"Active while Wirebody is open"** to keep this expectation honest.

## What this means for agents

MCP tools should expect transient `Unreachable` errors when the user steps away from the phone. The MCP server returns a clear error message in that case:

```json
{
  "code": "Unreachable",
  "message": "Cannot reach Wirebody at http://192.168.1.244:5606. Is the iOS app foregrounded and the LAN server enabled?"
}
```

The right pattern for agents: surface the message verbatim to the user. They'll know what to do.

## Local Network permission

The first time the LAN server starts on a given iOS install, iOS shows the **Local Network usage** permission sheet:

> "Wirebody" Would Like to Find and Connect to Devices on Your Local Network.

Allowing this lets the app bind a TCP listener. Denying it makes `NWListener.start` fail silently — the Wirebody Settings row will show **"Off"** with no specific error.

You can re-grant at iOS Settings → Privacy & Security → Local Network → Wirebody.

## Port and addressing

Default port is **8080**, configurable from 1024-65535. The displayed URL uses the iPhone's primary non-loopback IPv4 (`en0` Wi-Fi interface) — typically a `192.168.x.x` or `10.x.x.x` address.

If your home network uses different subnets for different SSIDs (guest vs main, 2.4 GHz vs 5 GHz on some routers), make sure the agent's host is on the same subnet as the iPhone. Tailscale, VPNs, or other overlay networks can route around the same-subnet requirement but require Bonjour / mDNS to traverse, which most do not handle out of the box.

## TLS

The LAN server speaks **plaintext HTTP**, not HTTPS. The traffic stays on your home Wi-Fi behind a router NAT, and managing a TLS cert chain on an iPhone-hosted server is friction not worth paying. If you need TLS (e.g., reverse-proxying through Tailscale Serve), set it up at the proxy layer.
